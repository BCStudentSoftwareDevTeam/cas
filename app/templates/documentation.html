Checklist

Are you a Code Reviewer or Submitter? Read your responsibilities first!!! Click one:


Note: before going through this checklist, run ./setup.sh to install required packages for codeReview.

    HOW TO BE A GOOD SUBMITTER?

  • In your PR description, provide the why (context!). What problem is your code trying to solve?
  • Expect Conversation
  • Accept Defeat
  • Use Automated Tools
  • Submit In Progess Work
  • Review Your Own Code
Submit WIP/Work-In-Progress Pull Request if you are working on bigger features even if when you're not completely done with your issue. Notify reviewer that your code is 30%-50% done. This makes the reviewer's job easier.
    Feedbacks to expect after submitting WIP pull request:
  • Architectural Issues
  • Problems with overall design
  • Design Pattern Suggestions
    Things to keep in mind:
  • Solving Multiple Problems? Break them into multiple branch/pull requests for ease of review.
  • If you wrote more than 500 lines of code, break it up and make multiple pull requests.
  • Documentation and Comments
  • Small Optimization
  • Talk to reviewers if you want to talk about Code Review process. Don't take feedback personally. It's an opportunity for growth.
    Feedbacks to expect after this point:
  • Nit picks
  • Variable Names
  • Documentation and Comments
  • Small Optimization

    HOW TO BE A GOOD REVIEWER:

  • BE OBJECTIVE: "This method is missing a docstring" INSTEAD OF "You forgot to write a docstring"
  • ASK QUESTIONS DON'T GIVE ANSWERS "Would it make more sense if..? "Did you think about ..?
  • OFFER SUGGESTIONS: "It might be easier to ...", "We tend to do it this way..."
  • AVOID THESE TERMS: simply, easily, just, obviously, well actually
  • HAVE CLEAR FEEDBACK:
    >Strongly support your opinions
    >Share How & Why
    >Link to supporting documentation, blog post, or stackoverflow examples
  • COMPLIMENT GOOD WORKS AND GREAT IDEAS
  • DON'T BE A PERFECTONIST
    >The goal is better code, not "exactly the code you would have written"
  • IT'S OK TO NIT-PICK
    >Syntax Issues
    >Spelling Errors
    >Poor variable names
    >Missing Corner Cases > Specify: Are your nitpicks blocking merge
    Save the nit-picks for last, after any pressing architechture, design or other large scale issues have been addressed.
  • AVOID BURN OUT: MAXIMUM 200-400 lines at a time
  • COMPLETE PR REVIEW AS SOON AS POSSIBLE
  • TRY TO PAIR REVIEW IF YOU ARE NEW TO THIS
  • YOU BECOME BUCH BETTER PROGRAMMER BY PARTICIPATING IN CODE REVIEWS
Source: Code Review for "Pythonistas"
